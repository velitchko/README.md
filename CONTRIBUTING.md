# Contributing to README.md

Thank you for your interest in contributing! This repository collects practical resources for reproducible visualization research, and community contributions are what make it useful. All skill levels and backgrounds are welcome.

## Ways to Contribute

### Tips & Tricks

Add a tip to [`tips-and-tricks/`](tips-and-tricks/) if you have a quick, actionable habit or workaround that has helped you make your own research more reproducible. Good tips are:

- Concrete and immediately applicable (not general advice)
- Grounded in a real problem you encountered
- Short enough to read in 30 seconds

Open a pull request adding your tip to the relevant section in `tips-and-tricks/README.md`, or [open an issue](../../issues) if you'd prefer to suggest it without writing it up yourself.

### Awesome Reproducibility Resources

Add an entry to [`awesome-reproducibility-resources/`](awesome-reproducibility-resources/) if you know of a tool, platform, paper, or template that is genuinely useful for reproducible visualization research. Each entry should follow the existing format:

```
- **[Name](url)**: What it is and *when a visualization researcher would reach for it*.
```

Explain the specific situation where this resource is the right choice.

### Tutorial Materials

Suggest improvements or additions to the hands-on materials in [`tutorial-materials/`](tutorial-materials/) by opening an issue. If you have example files, scripts, or templates that illustrate a reproducibility practice, we would love to include them.

### Bug Reports & Corrections

If you spot a broken link, an outdated tool recommendation, incorrect information, or a typo, please [open an issue](../../issues) or submit a pull request directly.

### Website

The tutorial website lives in [`website/`](website/) and is built with React + Vite. To run it locally:

```bash
cd website
npm install
npm run dev
```

Pull requests for content corrections and accessibility improvements are welcome. For larger UI changes, please open an issue first to discuss.

## Guidelines

- **One contribution per pull request**: keep PRs focused so they are easy to review.
- **Be specific**: "use Docker when your project has complex system-level dependencies" is more useful than "use Docker for reproducibility".

## Opening a Pull Request

1. Fork the repository and create a branch from `main`.
2. Make your changes.
3. Open a pull request with a short description of what you added and why it is useful.

If you are unsure whether something fits, open an issue first: we are happy to discuss before you invest time writing it up.

## Code of Conduct

Please be respectful and constructive in all interactions. We want this to be a welcoming space for researchers at all career stages.
